# Codex Goal: 모바일/PC 웹게임 「마왕성 러시」 완성 구현

너는 모바일 웹게임 개발자이자 게임 기획자다.  
현재 저장소에 「마왕성 러시」를 바로 플레이 가능한 웹게임으로 구현하라.

목표는 기획서 작성이 아니라 **사용자가 배포만 하면 즉시 플레이 가능한 완성본 구현**이다.  
작업 완료 전까지 반드시 직접 실행, 테스트, 빌드 검증을 진행하라.

이 문서는 단일 최종 지시문이다. 문서 안의 모든 내용은 아래 기준으로 일관되게 구현한다.

---

## 1. 최종 완성 조건

최종 결과물은 아래 조건을 만족해야 한다.

1. 모바일 웹브라우저와 PC 웹브라우저에서 바로 실행 가능해야 한다.
2. 게임은 180초 완주형이 아니라 **끝없이 이어지는 무한 침공 생존/점수 게임**이다.
3. **120초마다 침공 사이클이 바뀐다.** 게임은 초기화되지 않고 다음 침공으로 이어진다.
4. 코어 HP가 0이 될 때만 결과 화면을 표시한다.
5. 120초, 180초, 3분 도달만으로는 종료하지 않는다.
6. 뒤로 갈수록 난이도는 완만하게가 아니라 **팍팍팍 체감될 정도로 급격히 상승**해야 한다.
7. 초반은 배우기 쉽지만, 2차 침공부터는 선택을 잘못하면 빠르게 무너질 정도로 어렵게 만든다.
8. 사용자의 선택은 빌드 방향, 난이도, 보상 구조, 생존 방식, 결과 문구에 실제로 영향을 줘야 한다.
9. 어떤 선택을 했는지 HUD, 전투 화면, 이펙트, 결과 화면에서 체감되어야 한다.
10. 게임 시작 후 3초 안에 용사 물량이 보인다.
11. 게임 시작 후 10초 안에 처치, 골드, 영혼 흡수, 점수 상승, 콤보 표시가 발생한다.
12. 사용자가 아무것도 누르지 않아도 게임은 자동으로 진행된다.
13. 카드 선택, 건설 선택, 위험 보상 선택은 방치 시 **무작위 자동 선택**된다.
14. 모든 주요 버튼과 선택지는 상태바 또는 타이머를 가진다.
15. 카드 선택창은 화면 중앙에 표시한다.
16. 선택 중에는 뒤의 전투 시간, 적 이동, 시설 공격, 파티클 진행이 매우 느려져 사용자가 고민할 시간을 갖는다.
17. 선택 타이머 자체는 계속 진행된다.
18. 스킬과 보상 선택지가 너무 자주 뜨면 안 된다. 선택 하나의 효과를 크게 만들고, 선택 주기는 적당히 띄운다.
19. 마왕 스킬은 너무 빨리 차면 안 된다. 초반에는 1회 사용까지 시간이 걸리고, 후반 위험도가 높을수록 더 자주 쓰게 만든다.
20. 스킬은 다양해야 한다. 최소 4종 이상 구현하거나, 기본 스킬 1종에 선택으로 파생 효과가 붙는 구조를 구현한다.
21. 스킬, 피격, 처치 인터랙션은 다양해야 한다. 넉백, 감속, 기절, 끌어당김, 폭발, 보호막, 저주, 출혈 같은 상태 변화가 화면에 보여야 한다.
22. 사용자가 켜고 끌 수 있는 **2배속/3배속 버튼**을 제공한다. 켜면 전투, 성장, 스폰, 선택 타이머까지 모두 2배/3배 빠르게 진행된다.
23. PC와 모바일 모두 세로 스크롤 없이 한 화면에서 플레이되어야 한다.
24. 서버 저장, 로그인, 온라인 랭킹, 결제, 광고는 구현하지 않는다.
25. 외부 이미지, 외부 사운드, 외부 폰트, CDN 에셋을 가져오지 않는다.
26. 필요한 모든 그래픽은 직접 만든 SVG, CSS, Canvas 드로잉, 절차적 픽셀아트로 구현한다.
27. 빌드 결과물이 외부 네트워크 에셋 없이 동작해야 한다.
28. 테스트 코드와 수동 확인을 통해 실제 플레이 가능성을 검증하라.

## 2. 기술 구현 기준

현재 저장소 구조를 먼저 확인하라.

- 빈 저장소이거나 실행 가능한 웹앱 구조가 없으면 Vite + TypeScript 기반으로 최소 프로젝트를 직접 구성한다.
- React가 이미 있으면 React를 사용한다.
- React가 없고 프로젝트가 비어 있다면 React + TypeScript + Vite로 구성해도 된다.
- 백엔드는 만들지 않는다.
- 서버 저장은 만들지 않는다.
- 로그인은 만들지 않는다.
- 온라인 랭킹은 만들지 않는다.

권장 구조:

```txt
/
├─ package.json
├─ index.html
├─ src/
│  ├─ main.tsx 또는 main.ts
│  ├─ game/
│  │  ├─ constants.ts
│  │  ├─ engine.ts
│  │  ├─ balance.ts
│  │  ├─ scoring.ts
│  │  ├─ entities.ts
│  │  └─ assets.ts
│  ├─ components/
│  │  ├─ GameView.tsx
│  │  ├─ Hud.tsx
│  │  ├─ BottomControls.tsx
│  │  ├─ ChoiceModal.tsx
│  │  ├─ ResultScreen.tsx
│  │  └─ PixelArt.tsx
│  ├─ styles/
│  │  └─ game.css
│  └─ tests/
│     ├─ scoring.test.ts
│     ├─ balance.test.ts
│     └─ gameFlow.test.ts
├─ tests/
│  └─ e2e/
│     └─ game.spec.ts
├─ vite.config.ts
└─ playwright.config.ts
```

필수 npm scripts:

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview",
    "test": "vitest run",
    "test:e2e": "playwright test"
  }
}
```

Playwright 설치나 브라우저 실행이 현재 환경에서 불가능하면 그 사실을 명확히 기록하고, 가능한 범위에서 Vitest와 빌드는 반드시 통과시켜라.

## 3. 게임명과 장르

게임명: **마왕성 러시**

장르:

- 세로형 자동 전투 던전 성장 게임
- 초고속 타이쿤형 디펜스
- 무한 침공 생존 점수 게임

한 줄 설명:

> 마왕성을 폭풍 성장시키며 120초마다 강해지는 왕국 침공을 끝없이 버티는 세로형 웹 게임.

## 4. 화면 방향과 레이아웃

세로형 모바일 화면을 기본으로 한다.  
PC에서도 중앙 정렬된 세로형 게임 화면으로 동일하게 플레이 가능해야 한다.

필수 레이아웃 기준:

- `html`, `body`, `#root`는 화면 높이에 고정한다.
- 일반 플레이 중 브라우저 세로 스크롤이 생기면 실패로 본다.
- 게임 컨테이너 최대 너비: 430px 전후
- 모바일 safe-area 대응
- 전투 영역은 남는 높이에 맞춰 줄어들거나 늘어난다.
- 터치 버튼은 최소 44px 이상을 우선하되, 작은 화면에서는 1화면 유지가 더 중요하다.
- 배경은 어두운 마왕성 분위기
- 그래픽은 귀여운 픽셀풍
- CSS에 `image-rendering: pixelated` 적용 가능

화면 구조:

```txt
┌────────────────────────┐
│ 상단 HUD                │
│ 침공/시간/점수/난이도    │
│ 진행도/위험도/코어 HP    │
│ 빌드 방향/최근 선택 영향 │
├────────────────────────┤
│ 상단 침입문              │
│ 용사 스폰 구역           │
│ 전투 구역                │
│ 데미지/상태/사망 연출     │
│ 골드·영혼 흡수 연출       │
├────────────────────────┤
│ 마왕성 시설 구역          │
│ 소굴/함정/제단/미끼/특수  │
├────────────────────────┤
│ 하단 조작 UI             │
│ 스킬/2배속,3배속/이벤트 타이머  │
└────────────────────────┘
```

선택 모달:

- 화면 정중앙에 뜬다.
- 배경 전투는 보이되 어둡게 처리한다.
- 선택 중 전투는 매우 느려진다.
- 선택지 3개는 모바일에서도 잘 읽히게 한다.
- 선택지에는 효과와 대가를 짧게 표시한다.

## 5. 에셋 제작 규칙

외부에서 이미지를 다운로드하지 마라.  
Unsplash, itch.io, OpenGameArt, 아이콘 사이트, 구글 이미지, CDN, 외부 폰트 사용 금지.

필요한 에셋은 아래 방식 중 하나로 직접 제작하라.

1. 인라인 SVG
2. CSS 픽셀아트
3. Canvas 절차적 드로잉
4. 직접 작성한 작은 SVG 파일
5. 이미지 생성 도구가 프로젝트 환경에 명시적으로 제공되어 있다면, 직접 생성한 이미지를 저장소에 포함

필수 시각 요소:

- 초보 용사
- 궁수
- 사제
- 기사
- 왕국 정예병
- 고블린 소굴
- 가시 함정
- 저주 제단
- 보물 미끼
- 마왕 코어
- 상단 침입문
- 골드 파티클
- 영혼 파티클
- 데미지 숫자
- 콤보 표시
- 상태이상 표시
- 스테이지 전환 배너
- 러시 경고 연출
- 결과 화면 보상 상자

사운드는 필수는 아니다.  
넣는다면 외부 파일을 쓰지 말고 WebAudio로 간단한 효과음을 절차적으로 생성하라.

## 6. 핵심 게임 루프

게임은 아래 흐름으로 자동 진행된다.

1. 용사들이 화면 위에서 아래로 침입한다.
2. 마왕성 시설들이 자동으로 공격한다.
3. 용사가 죽으면 골드, 영혼, 점수, 콤보가 발생한다.
4. 골드와 영혼 파티클은 마왕 코어로 빨려 들어간다.
5. 골드는 자동 성장과 건설에 사용된다.
6. 일정 시간마다 업그레이드 카드 3장이 등장한다.
7. 일정 시간마다 건설 선택 기회가 등장한다.
8. 일정 시간마다 위험 보상 이벤트가 등장한다.
9. 스킬 게이지가 차면 사용자가 직접 스킬을 발동할 수 있다.
10. 시간이 지날수록 용사 수와 강도가 증가한다.
11. 120초마다 침공 사이클이 바뀌며 난이도와 보상 배율이 상승한다.
12. 코어 HP가 0이 되면 결과 화면을 보여준다.

사용자가 아무것도 하지 않아도 게임은 진행되어야 한다.  
단, 고득점과 장기 생존은 선택을 잘해야 가능해야 한다.

## 7. 시간 구조와 무한 침공

한 침공 사이클은 120초다.

| 국면 | 시간 | 이름 | 특징 |
| --- | ---: | --- | --- |
| Phase 1 | 0~20초 | 정찰대 침입 | 기본 용사 다수 등장 |
| Phase 2 | 20~50초 | 모험가 범람 | 파티 단위 침입 증가 |
| Phase 3 | 50~90초 | 왕국 공세 | 기사, 사제, 정예 등장 |
| Phase 4 | 90~120초 | 토벌대 러시 | 대량 침공, 사이클 말 폭주 |

120초가 되면:

- 게임을 끝내지 않는다.
- 시간을 0으로 리셋하는 연출을 하지 않는다.
- 내부적으로 다음 침공 사이클로 이어간다.
- HUD는 `침공 2차`, `침공 3차`처럼 표시한다.
- 용사 HP, 속도, 스폰율, 정예 비율, 코어 피해량이 크게 오른다.
- 보상 배율도 올라가지만 위험 상승폭이 더 커야 한다.

난이도 상승 기준:

- 1차 침공: 초반 학습용. 방치해도 어느 정도 버틴다.
- 2차 침공: 선택을 잘못하면 코어가 흔들리기 시작한다.
- 3차 침공: 고득점 위험 선택을 많이 했다면 빠르게 터질 수 있다.
- 4차 침공 이상: 빌드 시너지가 없으면 생존이 거의 불가능해야 한다.

권장 난이도 공식 예시:

```txt
difficulty =
1
+ (침공차수 - 1) x 0.55~0.75
+ 현재국면 보정
+ 러시 보정
+ 탐욕/문개방/정예유인 선택 압력
+ 방치/자동선택 누적 패널티 일부
```

뒤로 갈수록 체감 상승이 약하면 실패다.  
특히 120초 이후부터는 적이 단순히 조금 늘어나는 수준이 아니라, 화면 밀도와 코어 압박이 분명히 달라져야 한다.

## 8. 초반 도파민 설계

게임은 빈 화면에서 천천히 시작하면 안 된다.

시작 시 기본 보유:

| 시설 | 시작 상태 |
| --- | --- |
| 고블린 소굴 | Lv1 |
| 가시 함정 | Lv1 |
| 마왕 코어 | 기본 장착 |

첫 10초 안에 반드시 발생해야 하는 것:

- 용사 8~15명 등장
- 첫 용사 처치가 2초 안에 발생
- 골드 드롭 발생
- 영혼 흡수 연출 발생
- 점수 팝업 발생
- 콤보 표시 발생
- 첫 자동 성장 또는 첫 업그레이드 예고 발생

초반이 쉬운 것은 허용하지만, 그 쉬움이 120초 이후까지 이어지면 실패다.

## 9. 용사 구현

용사는 화면 위에서 아래로 내려온다.  
x 위치는 여러 레인 중 랜덤 또는 가중 랜덤으로 정한다.

MVP 용사 5종:

| 용사 | 역할 | 기본 점수 |
| --- | --- | ---: |
| 초보 용사 | 기본 적 | 10 |
| 궁수 | 빠른 적, 낮은 체력 | 15 |
| 사제 | 주변 용사 회복 | 30 |
| 기사 | 높은 체력 | 50 |
| 왕국 정예병 | 후반 강적 | 150 |

추가 권장 용사:

| 용사 | 역할 |
| --- | --- |
| 방패병 | 받는 피해 감소, 앞줄 보호 |
| 폭탄병 | 코어 도달 시 큰 피해 |
| 깃발병 | 주변 이동속도 증가 |
| 마법사 | 시설 일시 약화 또는 보호막 부여 |

용사가 마왕 코어에 도달하면:

- 코어 HP 감소
- 침공 위험도 상승
- 작은 화면 흔들림
- 피격 이펙트 표시
- 용사는 사라진다

피격 다양화:

- 약한 적: 작은 충돌 이펙트
- 기사/정예: 큰 충격파
- 폭탄형 적: 폭발
- 마법형 적: 코어에 디버프 표시

## 10. 시설 구현

초기 MVP 시설은 최소 4종이다.

| 시설 | 역할 | 연출 |
| --- | --- | --- |
| 고블린 소굴 | 단일 공격 | 고블린이 튀어나와 공격 |
| 가시 함정 | 경로 피해 | 용사가 밟으면 가시가 튀어 오름 |
| 저주 제단 | 범위 지속 피해 | 보라색 장판 |
| 보물 미끼 | 감속 및 골드 증가 | 용사가 느려지고 골드 연출 강화 |

추가 시설 또는 파생 시설을 최소 2개 이상 고려하라.

| 시설 | 역할 |
| --- | --- |
| 화염 항아리 | 폭발 광역 피해 |
| 뼈 감옥 | 짧은 속박/기절 |
| 악마 포탑 | 원거리 단일 고화력 |
| 피의 문장 | 코어 보호막 또는 흡혈 |

시설 슬롯은 고정 슬롯 방식으로 구현한다.  
자유 배치는 구현하지 않는다.

## 11. 선택 시스템

선택의 목적은 사용자가 성장 방향을 선택했다고 느끼게 만드는 것이다.  
무엇을 골라도 비슷하게 진행되면 실패다.

선택 종류:

1. 업그레이드 카드
2. 건설/시설 선택
3. 위험 보상 선택
4. 스킬 변형 선택
5. 침공 대응 선택

공통 동작:

- 선택지 3개 표시
- 중앙 모달 표시
- 자동 선택 타이머 표시
- 선택 중 전투는 매우 느리게 진행
- 선택 타이머는 계속 진행
- 사용자가 선택하면 즉시 적용
- 방치하면 무작위 자동 선택
- 무작위 자동 선택 결과는 HUD나 토스트로 알려준다

선택 주기:

- 첫 카드: 16~22초 사이
- 이후 카드: 28~36초마다
- 첫 건설: 35~45초 사이
- 이후 건설: 55~70초마다
- 위험 보상: 각 120초 사이클에 1~2회
- 스킬 변형 선택: 너무 자주 나오지 않게 1차 침공 중 1회, 이후 침공마다 1회 정도

선택이 너무 자주 떠서 게임 흐름을 끊으면 실패다.  
대신 선택 하나의 효과는 크고 눈에 띄어야 한다.

## 12. 업그레이드 카드

기본 카드 9종:

| 카드 | 효과 |
| --- | --- |
| 고블린 증원 | 고블린 소굴 공격력과 공격 속도 증가 |
| 가시 강화 | 가시 함정 피해와 코어 근처 방어 증가 |
| 저주 확산 | 저주 제단 범위와 지속 피해 증가 |
| 탐욕의 계약 | 골드 획득량 증가, 침공 압력 증가 |
| 침입문 개방 | 용사 수와 보상 증가, 난이도 상승 |
| 보물 미끼 강화 | 감속 효과와 골드 보너스 증가 |
| 영혼 수확 | 점수와 스킬 게이지 획득 증가 |
| 마왕 분노 | 스킬 충전 효율 증가 |
| 미믹 출현 | 폭발 피해 또는 랜덤 보상 |

추가 카드 예시:

| 카드 | 효과 |
| --- | --- |
| 피의 보호막 | 코어 보호막 획득, 골드 보상 감소 |
| 뼈 감옥 | 일정 주기 속박 발동 |
| 악마 징병 | 고블린 계열 강화, 코어 피해량 증가 위험 |
| 화염 세례 | 폭발 이펙트 강화, 사제 등장률 증가 |
| 정예 사냥 | 정예 점수 증가, 정예 출현률 증가 |
| 세금 징수 | 즉시 골드 획득, 다음 웨이브 강화 |
| 검은 제물 | 코어 HP 일부 소모, 강한 버프 획득 |
| 시간 왜곡 | 짧은 감속 필드 생성, 보상 감소 |

카드 선택 시 반드시 눈에 보이는 변화가 있어야 한다.

예시:

- 고블린 증원: 고블린 아이콘 추가 또는 공격 이펙트 증가
- 가시 강화: 가시가 더 크게 튀어 오름
- 저주 확산: 보라색 장판 확대
- 탐욕의 계약: 골드 파티클 증가와 위험도 상승 표시
- 침입문 개방: 상단 문이 커지고 즉시 추가 웨이브 유입
- 보물 미끼 강화: 미끼가 빛나고 적 이동 둔화
- 영혼 수확: 영혼 파티클 증가
- 마왕 분노: 스킬 버튼 발광 강화
- 미믹 출현: 미믹 상자가 등장해 폭발 또는 보상 지급

## 13. 건설 선택

건설 후보:

| 시설 | 역할 |
| --- | --- |
| 고블린 소굴 | 단일 공격 |
| 가시 함정 | 경로 피해 |
| 저주 제단 | 광역 지속 피해 |
| 보물 미끼 | 감속 및 골드 증가 |
| 화염 항아리 | 광역 폭발 |
| 뼈 감옥 | 속박/기절 |
| 악마 포탑 | 정예 저격 |
| 피의 문장 | 보호막/흡혈 |

건설 선택은 빌드 방향을 분명히 바꿔야 한다.

빌드 예시:

- 고블린 학살형: 빠른 단일 처치, 초중반 강함, 후반 물량에 약함
- 함정 방어형: 코어 근처 방어 강함, 보상 낮음
- 저주 장판형: 물량 처리 강함, 사제/정예 압력 증가
- 미끼 제어형: 감속과 골드 수급 강함, 폭탄형 적에 취약
- 폭발 고득점형: 위험하지만 점수 폭발
- 보호막 생존형: 오래 버티지만 성장 속도 낮음

## 14. 마왕 스킬

스킬은 너무 빨리 차면 안 된다.  
초반에는 최소 35~55초 정도 플레이해야 첫 사용이 가능하도록 조정한다.  
처치, 위험도, 카드 선택으로 빨라질 수는 있지만, 초반부터 난사되면 실패다.

기본 스킬 4종 이상을 구현하거나, 아래 중 1개 기본 스킬에 선택형 파생을 붙여도 된다.

| 스킬 | 효과 | 인터랙션 |
| --- | --- | --- |
| 마왕의 손아귀 | 적을 중앙으로 끌어당겨 피해 | 끌어당김, 큰 피해, 영혼 흡수 |
| 지옥불 낙인 | 넓은 폭발 피해 | 화염 폭발, 화상 지속 피해 |
| 시간 균열 | 일정 시간 적 이동 느려짐 | 화면 왜곡, 감속 |
| 피의 보호막 | 코어 보호막 생성 | 보호막 게이지, 피격 흡수 |
| 죽음의 칙령 | 정예 또는 앞줄 집중 처형 | 검은 표식, 즉사/큰 피해 |
| 뼈 감옥 | 적 무리 속박 | 속박 표시, 정지 |

UI:

```txt
마왕의 손아귀
████████░░ 82%
```

게이지가 100%가 되면:

- 버튼 테두리 발광
- 버튼이 커졌다 작아지는 애니메이션
- `사용 가능` 표시
- 터치 시 화면 전체 이펙트 발생

사용 시:

- 화면 흔들림
- 스킬별 고유 이펙트
- 다수 피해 숫자 표시
- 상태이상 아이콘 표시
- 처치 수 팝업
- 영혼 흡수 연출 강화
- 게이지 0으로 초기화

스킬 게이지 충전:

- 시간 경과로 아주 조금 충전
- 용사 처치 시 추가 충전
- 정예 처치 시 더 많이 충전
- 침공 위험도가 높으면 더 빨리 충전
- 카드 `마왕 분노`, `영혼 수확`으로 충전량 증가
- 2배속,3배속 상태에서는 충전도 2배속,3배속 흐름에 맞춘다

권장 충전 밸런스:

- 초반 시간 충전은 낮게 둔다.
- 처치 기반 충전도 너무 높게 잡지 않는다.
- 위험도가 낮은 안정 상태에서는 스킬이 천천히 찬다.
- 위험도가 높아질수록 구제 수단처럼 빨리 차지만, 무한 난사는 안 된다.

## 15. 피격과 상태 인터랙션

전투가 단순히 숫자만 줄어드는 느낌이면 실패다.  
아래 인터랙션을 여러 곳에 분산해서 구현하라.

필수 인터랙션:

- 피격 시 짧은 흰색 flash
- 강한 피격 시 넉백 또는 움찔
- 가시 피격 시 위로 튀는 느낌
- 저주 피격 시 보라색 틱 데미지
- 감속 상태 시 적 발밑 색 변화
- 기절/속박 상태 시 작은 아이콘 표시
- 코어 피격 시 화면 흔들림과 코어 균열/발광
- 정예 처치 시 큰 점수 팝업과 파티클 증가
- 스킬 사용 시 일반 공격과 구분되는 화면 전체 반응

상태 예시:

| 상태 | 효과 |
| --- | --- |
| 감속 | 이동 속도 감소 |
| 속박 | 짧은 시간 이동 정지 |
| 기절 | 이동과 회복 중지 |
| 저주 | 지속 피해 |
| 화상 | 짧은 지속 피해 |
| 표식 | 받는 피해 증가 |
| 보호막 | 코어 피해 흡수 |

## 16. 위험 보상 이벤트

위험 보상 이벤트는 고득점과 생존을 갈라야 한다.

예시 이벤트:

```txt
왕국의 도발장

1. 무시한다
2. 침입문을 더 연다
3. 왕국 기사단을 유인한다
```

선택 결과:

| 선택 | 결과 |
| --- | --- |
| 무시한다 | 안정적으로 진행, 점수 기대값 낮음 |
| 침입문을 더 연다 | 용사 수 증가, 골드/점수 증가 |
| 왕국 기사단을 유인한다 | 강한 적 즉시 등장, 큰 점수 보너스 |
| 제물을 바친다 | 코어 HP 감소, 강력한 성장 |
| 성문을 봉한다 | 잠시 안정, 보상 감소 |

방치 시:

- 무조건 안전 선택이 아니라 무작위 자동 선택
- 자동으로 위험 선택이 걸릴 수 있어야 한다

이벤트 적용 시 화면에 짧은 토스트 메시지를 보여라.

## 17. 침공 위험도와 난이도

침공 위험도는 HUD에 항상 표시한다.

계산 예시:

```txt
침공 위험도 =
살아있는 용사 수
+ 코어 근처 용사 수 x 2
+ 정예 용사 수 x 3
+ 코어 피해 누적 보정
+ 현재 침공차수 보정
+ 위험 선택 누적 보정
```

0~100% 게이지로 표시한다.

난이도 상승은 강해야 한다.

각 침공차수 상승 시 최소 변화:

- 스폰율 대폭 증가
- 적 HP 증가
- 적 이동속도 증가
- 정예/기사/특수 적 비율 증가
- 코어 도달 피해 증가
- 사제 회복량 또는 회복 빈도 증가
- 위험 선택 누적 시 추가 강화

자동 보정은 게임을 너무 쉽게 만들면 안 된다.

허용되는 보정:

- 위험도가 너무 높으면 스킬 충전량 소폭 증가
- 골드 수급이 너무 낮으면 보상 소폭 증가
- 초반 첫 처치 보장

금지되는 보정:

- 플레이어가 아무 선택을 안 해도 계속 안정적으로 버티게 만드는 보정
- 후반 난이도를 몰래 낮추는 보정
- 코어가 죽기 직전에 무조건 살려주는 보정

## 18. 성장과 보상 속도

120초마다 판이 바뀌므로 성장과 보상은 너무 느리면 안 된다.  
하지만 선택지가 너무 자주 떠서 전투를 끊으면 안 된다.

권장 방향:

- 선택 빈도는 낮추고 선택 효과는 크게 한다.
- 골드/영혼/점수 보상은 120초 사이클에 맞춰 빠르게 오른다.
- 자동 성장은 존재하되, 선택보다 약해야 한다.
- 고득점은 위험 선택을 감수해야 가능하다.
- 안정 선택만 하면 오래 버틸 수 있지만 점수 성장률이 낮아야 한다.

## 19. 점수 시스템

점수 이름:

> 약탈 점수

기본 공식:

```txt
약탈 점수 =
처치한 용사 수 x 10
+ 처치한 정예 용사 수 x 75
+ 획득 골드
+ 최대 콤보 x 25
+ 도달 누적 스테이지 x 1000
+ 최고 침공차수 보너스
+ 위험 선택 점수 배율
```

결과 화면은 코어 HP가 0이 되었을 때 표시한다.

등급:

| 점수 | 등급 |
| ---: | --- |
| 0~4,999 | 풋내기 마왕 |
| 5,000~9,999 | 동굴 지배자 |
| 10,000~19,999 | 마왕성 관리자 |
| 20,000~34,999 | 악몽의 마왕 |
| 35,000 이상 | 왕국 파괴자 |
| 70,000 이상 | 불멸의 마왕 |
| 120,000 이상 | 왕국 종말 |

점수 코드 형식:

```txt
RUN-{점수}-{처치수}-{최대콤보}-{최고침공차수}
```

상단 HUD 필수 정보:

- 현재 침공차수
- 현재 120초 사이클 국면
- 다음 국면 또는 다음 침공까지 남은 시간
- 현재 점수

예시:

2배속,3배속 버튼:

## 22. 성능 기준

모바일 웹게임이므로 과도한 DOM 노드를 만들지 마라.

권장:

- 전투 영역은 Canvas 또는 최적화된 DOM/SVG로 구현
- 데미지 숫자와 팝업은 개수 상한 적용
- 파티클 개수 상한 적용
- requestAnimationFrame 사용
- deltaTime 기반 업데이트
- 화면 밖 객체 정리
- 한 화면 적 수가 많아도 프레임이 크게 무너지지 않도록 처리

권장 상한:

- 활성 용사: 100~150명
- 파티클: 120개 이하
- 데미지 팝업: 40개 이하
- 이펙트: 50개 이하

상한에 도달하면 오래된 효과부터 제거하라.

## 23. 테스트와 디버그 모드

무한 게임이지만 테스트를 위해 디버그 시간 배속을 구현하라.

예시:

```txt
?debug=1&timeScale=20
?debug=1&timeScale=60
```

일반 접속에서는 반드시 1배속으로 동작해야 한다.  
디버그 배속과 사용자 2배속,3배속 토글은 별개로 동작하게 해도 된다.

필수 Unit Test:

1. 침공 사이클 계산
   - 0초: 침공 1차 Phase 1
   - 20초: Phase 2
   - 50초: Phase 3
   - 90초: Phase 4
   - 120초: 침공 2차 Phase 1
   - 240초: 침공 3차 Phase 1
   - 어떤 시간에도 시간 도달만으로 종료되지 않음

2. 난이도 계산
   - 2차 침공은 1차보다 확실히 어렵다.
   - 3차 침공은 2차보다 확실히 어렵다.
   - 위험 선택 누적 시 난이도가 추가 상승한다.

3. 선택 처리
   - 카드 선택을 하지 않으면 무작위 자동 선택되는지
   - 건설 선택을 하지 않으면 무작위 자동 선택되는지
   - 위험 보상 이벤트를 선택하지 않으면 무작위 자동 선택되는지
   - 선택 결과가 `choiceHistory` 또는 동등한 기록에 남는지

4. 선택 중 슬로우
   - 선택 모달이 열려 있을 때 전투 시간이 크게 느려지는지
   - 선택 타이머는 계속 줄어드는지

5. 스킬 충전
   - 초반 스킬이 너무 빨리 차지 않는지
   - 위험도 상승 또는 카드 선택으로 충전 속도가 증가하는지

6. 점수 계산
   - 처치 수, 정예 처치 수, 골드, 최대 콤보, 도달 스테이지, 최고 침공차수가 공식에 맞게 계산되는지

7. 점수 코드
   - `RUN-{점수}-{처치수}-{최대콤보}-{최고침공차수}` 형식인지 확인

필수 E2E 또는 통합 테스트:

1. 모바일 viewport에서 게임 화면이 로드된다.
2. PC viewport에서도 중앙 세로형 화면으로 로드된다.
3. 모바일/PC 모두 세로 스크롤이 생기지 않는다.
4. 시작 후 빠르게 용사가 등장한다.
5. 디버그 배속에서 10초 내 처치 수와 점수가 0보다 커진다.
6. 카드 선택 팝업이 중앙에 나타난다.
7. 선택 중 전투 슬로우 상태가 적용된다.
8. 카드 자동 선택이 무작위로 동작한다.
9. 2배속,3배속 버튼을 누르면 표시 상태가 바뀐다.
10. 120초 이후에도 결과 화면이 뜨지 않고 다음 침공으로 이어진다.
11. 스킬 버튼이 충전되고 사용 가능해진다.
12. 스킬 클릭 시 화면 상태와 적 상태가 변한다.
13. 코어 HP가 0이 되면 결과 화면이 나타난다.
14. 다시 도전 버튼 클릭 시 게임이 초기화된다.

테스트 환경상 Playwright 실행이 어렵다면:

- Vitest는 반드시 실행
- `npm run build`는 반드시 성공
- 수동 확인 내용을 최종 보고에 남겨라

## 28. 최종 목표

최종 결과물은 사용자가 별도 개발 개입 없이 아래만 수행하면 플레이 가능해야 한다.

```bash
npm install
npm run dev
```

또는 배포 시:

```bash
npm run build
```

완성 기준은 “코드가 있다”가 아니라 “무한 침공이 실제로 진행되고, 선택에 따라 빌드와 난이도와 결말이 달라지며, 코어가 무너질 때까지 플레이 가능한 웹게임”이다.
